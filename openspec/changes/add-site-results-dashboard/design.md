## Context

The existing `/admin` dashboard stores generic agent run data in `agent_runs`, `agent_run_steps`, and `agent_artifacts`. That model is intentionally execution-centric and works well for failures, retries, integration status, and artifact inspection. It is not ideal as the primary product view once crawling becomes automatic, because users will want to start from a site/URL and see the latest business result rather than a raw list of runs.

## Goals / Non-Goals

**Goals:**
- Provide a site-centric results surface that can be used independently from the operations dashboard.
- Persist latest site state and historical snapshots generated by RevenueAgent runs.
- Preserve links back to run detail pages for debugging.
- Keep the schema generic enough to support future analysis categories beyond SEO.
- Reuse existing admin token protection for all result pages.

**Non-Goals:**
- Building automatic crawl scheduling in this change.
- Building charts or trend analytics beyond showing historical snapshots.
- Replacing the existing operations dashboard.
- Changing RevenueAgent external API request/response contracts.

## Decisions

1. **Separate site result tables from run tables**

   Add `analyzed_sites`, `site_snapshots`, and `site_proposals`. `agent_runs` remains the execution log, while site tables become the product state. This avoids turning `/admin` into the source of truth for end-user review and leaves room for scheduled crawling later.

2. **Normalize by analyzed URL**

   A site record is keyed by normalized target URL, with domain stored for display/filtering. This is more precise than domain-only grouping when different paths or subdomains need separate results, while still making the domain visible.

3. **Store snapshot details as JSON**

   First-class columns cover common list/detail needs: status, score, domain, target URL, run ID, and timestamps. Diagnostics, summaries, and future analysis fields remain JSON to avoid schema churn when adding new analysis types.

4. **Route results under `/sites`**

   `/sites` becomes the site/result dashboard and `/admin` remains the operations dashboard. The pages link to each other, but each has a distinct purpose.

5. **Reuse token-based protection**

   Site result pages can contain proposal content and target data, so they use the same `ADMIN_TOKEN` policy as `/admin`. A shared helper keeps behavior consistent.

## Risks / Trade-offs

- [Risk] The initial schema duplicates some values from `agent_runs.summary_json`.
  → Mitigation: Keep duplicated fields limited to query/display indexes and keep the full run link for source detail.
- [Risk] URL normalization may not perfectly match future crawl grouping needs.
  → Mitigation: Store both normalized URL and domain, and keep generated IDs internal so grouping can evolve later.
- [Risk] Proposal content can make pages heavy.
  → Mitigation: Store proposal records separately and render the latest proposal first with historical records collapsed to metadata where possible.
